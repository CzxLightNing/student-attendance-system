## 一、项目概述

### 1.1 项目背景

传统课堂考勤依赖纸质点名或口头点名，存在效率低、易出错、数据难统计等问题。本系统旨在通过信息化手段解决这些痛点：教师可在课堂上即时生成签到码，学生通过 Web 端输入签到码完成签到，系统自动记录并实时展示签到结果，管理员可对班级、用户、考勤数据进行集中管理。

### 1.2 系统定位

基于 Django 框架的 **B/S 架构（Browser/Server）** Web 应用系统，采用 **MTV（Model-Template-View）** 架构模式，支持管理员、教师、学生三种角色协同使用。

### 1.3 项目目标

- 实现高效、防作弊的课堂签到流程
- 提供实时签到监控，教师无需手动刷新即可看到签到进展
- 支持 Excel 考勤数据导出，便于教学管理
- 具备完善的权限体系与安全防护机制

## 三、核心功能详解

### 3.1 三种角色权限体系

| 角色 | 核心权限 | 首页 |
|------|----------|------|
| 管理员 | 班级 CRUD、用户 CRUD、CSV 批量导入、账号审核、考勤总览 | 管理面板统计页 |
| 教师 | 生成签到码、实时签到监控、查看学生名单、导出 Excel | 我的班级列表 |
| 学生 | 输入签到码签到、查看签到历史、修改密码 | 签到页 |

### 3.2 管理员功能

#### 管理面板首页
- 展示班级总数、教师总数、学生总数、今日签到次数、待审核账号数
- 快捷操作入口：创建班级、添加用户、导入 CSV、考勤总览

#### 班级管理
- 增删改班级信息（名称、年级）
- 支持 CSV 批量导入班级，提供模板下载
- 文件上传安全校验：扩展名 `.csv`、MIME `text/csv`、大小 ≤ 5MB

#### 用户管理
- 按角色筛选（学生/教师）、按班级筛选学生
- 支持逐个添加用户，也支持 CSV 批量导入
- 批量导入时自动分配班级关联
- 管理员可重置任意用户密码（重置为初始密码 `12345678`）
- 管理员审核/激活用户自行注册的账号

#### 考勤总览
- 查看所有班级的签到统计汇总

### 3.3 教师功能（核心业务流程）

#### 签到码生成（核心功能）
- 教师选择一个班级 → 进入签到管理页
- 生成 **4 位随机数字** 签到码，同一班级同一时间仅一个有效码
- 支持设置有效时长：5 分钟 / 10 分钟 / 15 分钟 / 30 分钟 / 60 分钟，默认 10 分钟
- 生成新码时旧码自动失效；教师也可手动失效当前签到码
- 使用 NTP 时间（ntp.aliyun.com）校验过期，不依赖客户端本地时间

#### 实时签到监控（核心亮点）
- 页面分为「已签到」和「未签到」两个区域
- 前端 JavaScript 每隔 **3 秒** 自动轮询 `/attendance/teacher/code/{id}/status/` 接口
- 无需手动刷新页面，实时展示签到进展（已签到/未签到学生列表）
- 使用 `setInterval` 定时器 + `fetch` API 实现

#### 学生名单查看
- 教师可查看所管理班级的完整学生名单（学号、姓名、用户名）
- 教师无权修改班级和学生数据

#### 考勤历史
- 展示该班级所有历史签到码及其状态（有效/已失效）
- 每条签到码可「查看详情」（签到学生列表）和「导出 Excel」

#### Excel 导出
- 导出字段：学号、姓名、签到时间、签到状态（正常/迟到/缺勤）
- 使用 `openpyxl` 生成 `.xlsx` 文件，含表头样式（加粗、居中、边框、背景色）
- 支持按日期范围导出某班级的全部签到汇总

### 3.4 学生功能

#### 签到页
- 登录后首页展示签到码输入框
- 若当前班级有有效签到码，提示"当前班级有有效签到码，请及时签到"
- 输入 4 位签到码后 AJAX 提交，系统执行三重校验：
  1. 签到码是否存在且有效（未过期、未失效）
  2. 学生是否属于该签到码对应的班级
  3. 该学生是否已针对此签到码签到（防重复）
- 成功则记录并显示签到时间，失败则返回具体错误提示

#### 签到历史
- 查看个人所有签到记录，支持按日期范围筛选
- 展示签到码、班级、签到时间、签到状态

## 四、数据库设计

### 4.1 核心数据模型

系统包含 4 个核心模型，关系如下：

```
┌──────────┐         ┌──────────────┐
│   Class  │ 1─────N │  CustomUser  │
│  班级     │         │    用户       │
└──────────┘         └──────────────┘
      │                      │
      │ 1                    │ N (managed_classes, 仅教师)
      │                      │
      ▼                      ▼
┌────────────────┐  ┌──────────────────┐
│ AttendanceCode │  │ AttendanceRecord │
│   签到码        │1─N│    签到记录       │
└────────────────┘  └──────────────────┘
```

### 4.2 模型详解

#### CustomUser（自定义用户模型）
- 继承 Django `AbstractUser`
- 新增字段：`role`（角色）、`student_id`（学号）、`teacher_id`（工号）
- `student_class` ForeignKey → Class（学生所属班级）
- `managed_classes` ManyToMany → Class（教师管理的班级）
- `is_active` 用于账号审核/激活

#### Class（班级）
- `name`：班级名称（唯一）、`grade`：年级、`created_at`：创建时间

#### AttendanceCode（签到码）
- `class_group` ForeignKey → Class
- `code`：4 位数字签到码
- `created_by` ForeignKey → User（创建教师）
- `expires_at`：过期时间（基于 NTP 时间生成）
- `is_active`：是否有效（可手动失效）

#### AttendanceRecord（签到记录）
- UniqueConstraint：`(student, attendance_code)` 防止重复签到
- `status`：正常 / 迟到 / 缺勤
- `timestamp`：签到时间

### 4.3 数据库兼容性设计

- 所有模型 `db_table` 显式指定表名，确保 MySQL 迁移时名称一致
- 所有 `CharField` 显式指定 `max_length`
- 所有 `ForeignKey` 显式设置 `on_delete` 行为
- 关键查询字段添加 `db_index=True` 和联合索引
- `DateTimeField` 使用 `auto_now_add`，兼容 MySQL 5.7+

## 五、安全设计

### 5.1 认证安全

| 措施 | 实现方式 |
|------|----------|
| 密码哈希 | argon2-cffi，配置为 Django 默认哈希算法，严禁明文存储 |
| 登录锁定 | django-axes：5 次失败锁定 30 分钟，同时按用户名+IP 锁定 |
| 会话安全 | 会话超时 2 小时，Cookie 设置 HttpOnly + SameSite=Lax |
| CSRF 保护 | 所有 POST 请求包含 CSRF Token，AJAX 请求 header 带 X-CSRFToken |

### 5.2 文件上传安全

| 措施 | 说明 |
|------|------|
| 扩展名白名单 | 仅允许 `.csv` |
| MIME 类型校验 | 必须为 `text/csv` |
| 文件大小限制 | 最大 5MB |
| 重命名存储 | 使用时间戳重命名，防止路径遍历攻击 |

### 5.3 审计日志

使用 Python `logging` 模块记录关键安全事件：
- 用户登录（成功/失败）
- 密码修改
- CSV 数据导入
- 日志文件路径：`logs/` 目录

### 5.4 时间安全

签到码过期判断不依赖客户端本地时间，使用 `ntplib` 从阿里云 NTP 服务器 `ntp.aliyun.com` 获取标准时间。

## 六、技术亮点详述

### 6.1 实时签到监控（AJAX 轮询）

**问题**：教师生成签到码后，如何实时知道哪些学生已签到？

**解决方案**：
1. 前端使用 `setInterval` 每 3 秒调用 `/attendance/teacher/code/{id}/status/` 接口
2. 后端返回当前签到码的已签到/未签到学生列表（JSON 格式）
3. 前端根据返回数据动态更新 DOM，将学生卡片在「已签到」「未签到」区域之间移动

**技术要点**：
- 纯原生 JavaScript `fetch` API，无前端框架依赖
- 页面卸载时自动 `clearInterval` 清理定时器，避免内存泄漏
- 轮询接口返回轻量 JSON，网络开销极小

### 6.2 NTP 时间同步

**问题**：如果依赖服务器本地时间，系统时间不准会导致签到码提前或延后过期。

**解决方案**：
- 使用 `ntplib` 库从 `ntp.aliyun.com` 获取标准时间
- 签到码过期时间、签到记录时间戳均基于 NTP 时间计算
- 在 `settings.py` 中设置 `TIME_ZONE = 'Asia/Shanghai'` + `USE_TZ = True`

### 6.3 角色权限装饰器

**问题**：不同角色无权访问其他角色的页面。

**解决方案**：
- 自定义 `AdminRequiredMixin`、`TeacherRequiredMixin`、`StudentRequiredMixin`
- 继承 `UserPassesTestMixin`，检查 `request.user.role`
- 权限不足时重定向到角色对应首页并显示提示信息
- 所有视图通过 `LoginRequiredMixin` + 角色 Mixin 双重保护

### 6.4 CSV 批量导入

**问题**：管理员需要批量导入几十甚至上百个班级/学生/教师数据。

**解决方案**：
- 提供标准 CSV 模板下载（`classes_template.csv` / `students_template.csv` / `teachers_template.csv`）
- 上传时执行三重安全校验
- 导入时自动创建关联（学生自动关联班级、教师自动关联管理班级）
- 使用 `utf-8-sig` 编码读取，兼容 Excel 导出的 CSV 文件（含 BOM 头）

### 6.5 URL 命名空间

项目使用 Django URL 命名空间确保路由不冲突：
- `accounts:login` / `accounts:home` / `accounts:register`
- `attendance:student_home` / `attendance:teacher_class` / `attendance:generate_code`
- `management_app:home` / `management_app:class_list`

所有 `reverse()`、`redirect()`、`{% url %}` 均使用命名空间前缀。
